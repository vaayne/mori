Yes — and for SSH + tmux control mode + libghostty, I’d build it as a native tmux client with libghostty as the pane renderer, not as a traditional “terminal app that happens to run tmux.”

Here’s the plan I’d use.

1) Product shape for v1

Make the iOS app a tmux workspace client with:
	•	one active tmux session open at a time,
	•	native UI for sessions / windows / panes,
	•	one libghostty surface per visible pane,
	•	SSH only as the transport to a remote tmux server.

That model fits tmux control mode well: a control-mode client is a text protocol endpoint that accepts normal tmux commands and receives async notifications like %output, %window-add, and %sessions-changed. A control-mode client is attached to one session at a time, and receives live pane output for any pane in any window in that attached session.  ￼

So the right v1 UX is:
	•	browse all sessions globally,
	•	attach one session as the current workspace,
	•	manage windows/panes natively inside that session,
	•	show previews for other sessions, but don’t try to keep multiple sessions live at once on one phone screen.

2) Core architecture

Split the app into six layers:

A. SSHConnectionManager

One SSH connection per host/profile. Use SSH channel multiplexing, not multiple TCP connections. SwiftNIO SSH is designed exactly like this: one SSH connection can contain multiple child channels, and it supports shell/exec session channels plus auth and forwarding. It is also explicitly a programmatic SSH implementation, not a ready-made terminal client, which fits your custom tmux app well.  ￼

Recommended channel layout:
	•	Channel 1: control-mode channel running tmux -CC ...
	•	Channel 2: optional exec channel for bootstrap/probing, such as tmux -V, existence checks, or one-shot recovery commands
	•	Channel 3: optional SFTP later for config/log export, key install, etc.

B. TmuxControlClient

A byte-oriented parser for the control-mode stream:
	•	%begin / %end / %error command blocks
	•	async notifications
	•	%output / %extended-output
	•	recovery after disconnect

tmux says %output payload is not guaranteed UTF-8 and escapes control chars and backslashes using octal sequences, so this parser should stay byte-based, not String-based, until after unescaping.  ￼

C. TmuxModelStore

Your canonical app state:
	•	Host
	•	Session ($id, name, flags)
	•	Window (@id, name, index, flags)
	•	Pane (%id, geometry, active, preview, rendererRef)

Use tmux’s stable IDs ($, @, %) everywhere internally, not names or indexes. tmux explicitly recommends IDs because they are unique and stable for the life of the object, and the control-mode docs recommend -F formatted output for machine parsing.  ￼

D. GhosttyPaneRenderer

One libghostty-backed renderer per visible pane only.
	•	Feed decoded pane bytes into the correct renderer.
	•	Keep hidden panes as lightweight model objects with cached previews.
	•	Recreate surfaces lazily when panes become visible.

Ghostty positions libghostty as the embeddable core for terminal emulation, and notes it is already heavily used, but also says the API signatures are still in flux and it hasn’t tagged a standalone stable libghostty version yet. That means you should wrap it behind your own Swift façade and pin a Ghostty revision.  ￼

E. TmuxInputMapper

This is important: for v1, I would not make libghostty own the entire input path.

Instead:
	•	libghostty owns rendering + terminal state
	•	your app owns input mapping into tmux commands

Reason: control mode is command-oriented, not a PTY write stream. tmux already gives you input primitives like send-keys, send-keys -l, send-keys -H, and send-keys -M.  ￼

F. WorkspaceUI

Native iOS UI:
	•	session sidebar / drawer
	•	window strip or cards
	•	pane grid
	•	command palette
	•	search / preview / rename / kill confirmations

This is not optional fluff. tmux itself says output generated by tmux UI modes, such as copy mode or choose mode, is not sent to control-mode clients. So if you want great mobile UX, native UI is the correct approach anyway.  ￼

3) How to attach to tmux

Use a startup sequence like this:
	1.	Open SSH connection.
	2.	Probe tmux availability/version on a one-shot exec channel.
	3.	Open the control channel and run either:
	•	tmux -CC attach-session -t <session>
	•	or tmux -CC new-session -A -s <session>

I’d spike exec channel without PTY first, because control mode is text-only. If that turns out quirky on some hosts, fall back to a shell channel. I would validate this early before writing too much app code.

After attach:
	•	immediately request structured state with list-sessions -F ..., list-windows -F ..., list-panes -F ...
	•	then start live updates from notifications and %output. tmux explicitly recommends those list commands plus -F output for control-mode users.  ￼

4) Session sizing strategy

This is the biggest product decision.

tmux control mode gives you refresh-client -C to set the control client size. If you do not use it, control clients do not affect window sizing. If you do use it, tmux treats that control client like a normal sized client and it may affect window size.  ￼

So ship two modes:

Mode 1: Mirror mode

For solo use on a session.
	•	On attach, set client size using refresh-client -C <cols>,<rows>
	•	Best interactive fit on iPhone/iPad
	•	May affect shared tmux layout depending on the session’s sizing behavior

Mode 2: Inspect mode

For “don’t disturb my desktop.”
	•	Do not call refresh-client -C
	•	You view the session at whatever size tmux already has
	•	Good for checking logs / state / quick actions

I would ship Mirror mode as default on iPad and Inspect mode as an option. Later, if you want the cleanest multi-device story, add a mobile shadow session feature.

Also attach with the active-pane client flag where possible. tmux documents that this lets a client keep an independent active pane for cursor/command purposes, which is useful for your native pane-focus UI.  ￼

5) Output pipeline

For the active workspace session:
	1.	Parse control lines.
	2.	Route %output %pane ... to the pane with that %id.
	3.	Decode octal escapes.
	4.	Feed raw bytes into that pane’s libghostty surface.

Because control-mode output for panes is “exactly what the application running in the pane sent to tmux,” you are effectively rendering the pane’s terminal stream directly, which is exactly what you want.  ￼

Two rules matter:

Visible panes

Keep libghostty surfaces alive and feed bytes incrementally.

Hidden panes

Do not keep every pane “hot.”
tmux provides flow control specifically for control-mode clients:
	•	refresh-client -f pause-after=<seconds>
	•	%pause
	•	%continue
	•	refresh-client -A '%pane:continue'
	•	and even manual pause / off behavior.  ￼

So for v1:
	•	enable pause-after
	•	mark offscreen panes off
	•	when a pane becomes visible again, rehydrate from capture-pane

tmux’s capture-pane can print visible content or full history, and can include escape sequences with -e.  ￼

Recommended policy:
	•	pane preview cards: capture-pane -p -S -80 -E - -t %pane without styling
	•	full renderer rehydrate after pause/resume: capture-pane -p -e -S - -t %pane

6) Input pipeline

This is where most control-mode apps get messy. Keep it simple.

Text input

Use:
	•	send-keys -l -t %pane "<text>"

Special keys

Use tmux key names:
	•	Enter
	•	BSpace
	•	Tab
	•	Up
	•	Down
	•	Left
	•	Right
	•	C-c
	•	M-b
and so on with send-keys -t %pane ...

Raw ASCII control bytes / edge cases

Use:
	•	send-keys -H -t %pane <hex> <hex> ...

tmux documents -l for literal UTF-8 text and -H for hex ASCII characters.  ￼

Mouse

Defer full terminal mouse passthrough to phase 2.
tmux supports forwarding mouse events with send-keys -M, so you have a path later, but I would not make this a v1 dependency.  ￼

Pane focus

Pane taps should call native tmux commands like:
	•	select-pane -t %pane
	•	select-window -t @window
rather than trying to synthesize terminal mouse immediately.

This gives you a much more reliable v1 on touch devices.

7) Native UI features you should implement instead of depending on tmux UI

Because tmux copy/choose UI is not emitted to control-mode clients, implement these natively:
	•	session picker
	•	window switcher
	•	pane overview
	•	rename session/window
	•	create/kill/swap window
	•	split/resize pane
	•	search scrollback
	•	copy selection
	•	recent pane previews

This is not just “nice UX”; it’s the correct architectural answer for control mode.  ￼

8) Metadata syncing

Use two mechanisms together:

A. Event-driven updates

From tmux notifications:
	•	%sessions-changed
	•	%session-changed
	•	%window-add
	•	%window-close
	•	%window-renamed
	•	%window-pane-changed
and related events.  ￼

B. Structured refresh

After any important event, refresh the relevant scope with:
	•	list-sessions -F ...
	•	list-windows -F ...
	•	list-panes -F ...

And use refresh-client -B subscriptions for lightweight metadata that changes over time, because tmux supports format subscriptions and emits %subscription-changed at most once per second when values change.  ￼

Good uses for subscriptions:
	•	pane title
	•	current command
	•	activity/bell flags
	•	session name
	•	window title

9) libghostty integration plan

Since you already have a Mac app on libghostty, reuse that investment.

I’d create a shared internal package:
	•	GhosttyBridge
	•	TmuxCore
	•	SSHCore

GhosttyBridge

Wrap all C calls behind a narrow Swift API:
	•	create/destroy surface
	•	feed bytes
	•	resize
	•	theme/font config
	•	selection/search hooks if exposed

Ghostty’s own repo describes libghostty as the cross-platform embeddable core, but also says versioning/docs are still maturing. That’s exactly why you want your own wrapper boundary.  ￼

Remote/pipe backend

A Ghostty maintainer explicitly said a pipe-based backend is the intended direction for alternate input mechanisms and remote instances, and an iOS libghostty app author described getting libghostty running with a pipe-based backend plus SwiftNIO SSH on iOS.  ￼

That means your iOS app should use the same mental model:
	•	remote bytes come in from tmux control mode
	•	renderer consumes bytes
	•	renderer lifecycle is local
	•	transport is fully app-owned

Rendering loop

One iOS libghostty adopter reported needing CADisplayLink for acceptable GPU behavior on Apple’s newer platforms. I would follow that lead rather than improvising a timer-based redraw loop.  ￼

10) Build and packaging

Keep the iOS libghostty piece as a separately versioned artifact.

Ghostty’s build/docs say:
	•	each Ghostty version requires a matching Zig version,
	•	macOS builds need Xcode with both macOS and iOS SDKs installed,
	•	and the build system has explicit Darwin xcframework emission paths.  ￼

So I’d do:
	•	pin one Ghostty revision/tag
	•	build GhosttyKit.xcframework
	•	wrap it in your internal Swift package
	•	only update Ghostty intentionally, not casually

11) iOS lifecycle plan

Do not design around persistent live sockets in the background.

Apple’s guidance is that iOS generally suspends apps shortly after they move to the background, and there is no general-purpose background execution mechanism for arbitrary long-running work.  ￼

So your lifecycle should be:

Foreground
	•	maintain SSH control connection
	•	live render visible panes

App moving to background
	•	persist current workspace/session/window/pane IDs
	•	persist pane previews and UI state
	•	optionally request a short background task only to flush state cleanly

Foreground resume
	•	reconnect SSH
	•	reopen control mode
	•	restore active session
	•	refresh sessions/windows/panes
	•	rehydrate visible panes using capture-pane
	•	continue streaming live output

tmux is your persistence layer, not iOS background execution.

12) Phased implementation order

Phase 0 — spike

Prove three things only:
	•	SSH connect works
	•	tmux -CC works reliably on your chosen SSH channel type
	•	one pane can render %output in libghostty

Do not build any fancy UI yet.

Phase 1 — single-session workspace

Ship:
	•	host/profile management
	•	attach one session
	•	session/window/pane listing
	•	single visible window
	•	visible panes rendered with libghostty
	•	keyboard input via send-keys

Phase 2 — native tmux management

Add:
	•	create/rename/kill session/window
	•	split/close/select pane
	•	pane previews
	•	focus handling
	•	inspect/mirror sizing modes

Phase 3 — robustness

Add:
	•	flow control
	•	offscreen pane suspension
	•	reconnect/resync
	•	preview hydration
	•	state restoration
	•	host key management polish

Phase 4 — polish

Add:
	•	iPad sidebar + multi-column layout
	•	iPhone pane overview mode
	•	search across pane history
	•	copy/share/export
	•	optional mouse passthrough
	•	theming synced with your Mac app

13) Biggest risks to design around early

The three real risks are:

1. Input fidelity

Especially modified keys, IME/text insertion, and paste.
Mitigation: start with explicit TmuxInputMapper, not raw PTY emulation.

2. Shared-session sizing semantics

A mobile client can affect layout if you let it.
Mitigation: ship Mirror vs Inspect modes from day one.

3. Rehydrating paused/offscreen panes

Live streaming every pane all the time will waste CPU and battery.
Mitigation: flow control + capture-pane recovery. tmux already gives you the tools for this.  ￼

⸻

My concrete recommendation

Build v1 like this:
	•	SSH transport: SwiftNIO SSH
	•	tmux integration: one control-mode channel
	•	renderer: libghostty surfaces per visible pane
	•	input: native mapper to tmux commands
	•	workspace model: one attached session at a time
	•	recovery: reconnect + list-* + capture-pane
	•	mobile UX: native session/window/pane UI, not tmux copy/choose UI

That is the most practical version of “SSH + tmux control mode + libghostty” for iOS.
